# AI 协作与发版规则

这份文件用于约束 Coding AI 在进行新功能开发、修复问题、准备新版本时的默认工作方式。

## 项目速览

- 项目类型：Electron + Three.js 的 Windows 本地 FBX 预览工具。
- 入口文件：`main.js` 是 Electron 主进程，`index.html` 承载主要界面和渲染层逻辑。
- 本地库文件：`libs/` 下保存 Three.js、FBXLoader、OrbitControls 以及贴图加载器等前端依赖。
- 当前版本号以 `package.json` 为准，当前为 `1.0.5`。
- 开发环境：Node.js 22.12+、npm、Windows 10/11。

## 常用命令

```bash
npm install
npm start
npm run pack
npm run dist
```

- `npm start`：开发运行 Electron 应用。
- `npm run pack`：生成未安装的可执行目录。
- `npm run dist`：通过 electron-builder 生成 Windows 安装包和便携版。
- 当前已配置 `npm test`，会运行 `.mesh` 路径、UV 数值与撤销、3ds Max 操作模式、操作可发现性、FBX 拖入/拖出、v1.0.4 功能约束和 Electron 启动 smoke 检查；完成修改后仍应按风险补充 `npm run pack` 或手动验证。

## 本项目注意事项

- 不要提交 `node_modules/`、`dist/`、`FBX_Data/`、`fbx_cache/` 或日志/临时文件。
- `FBX_Data` 是运行期数据目录；小配置用于收藏、UV 元数据缓存和窗口状态，缩略图缓存优先放在 portable/exe 或安装目录同级的 `FBX_Data/fbx_cache`，安装目录不可写时才让用户选择位置。
- 修改 `index.html` 时注意它包含大量渲染层逻辑，尽量保持变更聚焦并手动验证关键 UI 流程。
- 修改 `libs/` 中第三方库前，先确认是否确实需要改 vendored 文件。

目标：

1. 不直接在 `main` 上开发
2. 开始改动前先创建合适的分支
3. 分支命名尽量和版本、任务类型有关
4. 修改后优先运行测试或验证命令
5. 完成后再准备合并或发布

---

## 默认原则

- `main` 始终代表当前稳定版本
- AI 默认不要直接在 `main` 上写功能代码
- 收到“做新功能”“做新版本”“加一个能力”“修一个 bug”这类任务时，应先创建分支，再开始修改
- 除非用户明确要求，否则不要直接发布，不要直接打 tag，不要直接改写 Git 历史

---

## 分支创建规则

### 1. 新功能

如果任务是新增功能，创建：

```text
feature/<简短功能名>
```

例子：

- `feature/login`
- `feature/export-pdf`
- `feature/user-profile`

### 2. 修复问题

如果任务是修 bug，创建：

```text
fix/<简短问题名>
```

例子：

- `fix/image-upload`
- `fix/login-timeout`

### 3. 紧急修复

如果任务是线上紧急修复，创建：

```text
hotfix/<简短问题名>
```

### 4. 与版本绑定的新版本开发

如果用户明确说“做下一个版本”或“基于当前版本做新版本”，优先读取当前版本号，然后创建：

```text
release/v<目标版本>
```

或者：

```text
feature/v<目标版本>-<简短功能名>
```

推荐规则：

- 如果这次是“准备一次版本发布”，用 `release/vX.Y.Z`
- 如果这次只是“为下个版本开发某个功能”，用 `feature/vX.Y.Z-功能名`

例子：

- `release/v1.3.0`
- `feature/v1.3.0-login`
- `fix/v1.3.1-payment-timeout`

## 版本号判断规则

如果项目中存在明确版本文件，AI 应优先从下面位置读取版本号：

1. `package.json`
2. `pyproject.toml`
3. `Cargo.toml`
4. `pom.xml`
5. `build.gradle` 或 `build.gradle.kts`
6. `.csproj`
7. `VERSION`
8. `README` 中明确写出的版本号

如果找到当前版本号，例如 `1.2.3`，则默认规则是：

- 修 bug 且准备小修复版：建议目标版本 `1.2.4`
- 新功能但不破坏兼容：建议目标版本 `1.3.0`
- 破坏性改动：建议目标版本 `2.0.0`

如果无法可靠判断版本号：

- 不要伪造版本号
- 改用普通分支命名，例如 `feature/login`

## 开始工作前的 Git 流程

AI 在开始改代码前，默认执行以下思路：

1. 检查当前是否为 Git 仓库
2. 检查当前分支
3. 检查工作区是否有未提交改动
4. 如果有用户未说明的脏改动，不要覆盖它们
5. 优先从当前最新的 `main` 创建新分支

理想流程：

```bash
git checkout main
git pull origin main
git checkout -b feature/login
```

如果本地已有未提交改动，AI 应先提醒风险，再决定是否继续，而不是直接切分支乱操作。

---

## 修改后的验证规则

AI 完成修改后，默认要尝试做至少一种验证。

优先顺序：

1. 项目已有测试命令时，运行测试
2. 没有完整测试时，运行最小可行验证
3. 如果连验证命令都没有，明确说明未能自动验证

常见验证包括：

- `npm test`
- `pnpm test`
- `pytest`
- `cargo test`
- `go test ./...`
- `dotnet test`
- 构建命令，如 `npm run build`
- lint 命令，如 `npm run lint`

如果用户说“先做，不用跑测试”，才可以跳过。

---

## AI 的默认执行策略

当用户说：

- “帮我做一个新版本”
- “加一个新功能”
- “修这个 bug”
- “做一下这个需求”

AI 默认应该这样做：

1. 先识别项目版本号和测试方式
2. 决定合适的分支名
3. 创建分支
4. 再开始改代码
5. 修改后运行测试或验证
6. 总结结果，并说明当前分支名

---

## 向用户汇报时必须包含的信息

AI 完成任务后，建议在总结里明确写出：

1. 当前工作分支名
2. 做了哪些修改
3. 跑了哪些测试或验证
4. 是否适合合并到 `main`
5. 是否建议打 tag 或创建 release

## 示例

### 示例 1：做一个新功能

项目当前版本：`1.2.3`

用户说：

- “帮我加一个登录功能”

AI 默认建议：

- 分支名：`feature/v1.3.0-login`

如果项目还没有明确版本管理，也可以退化成：

- `feature/login`

### 示例 2：修一个 bug

项目当前版本：`1.2.3`

用户说：

- “修一下支付超时问题”

AI 默认建议：

- 分支名：`fix/v1.2.4-payment-timeout`

或者简单一些：

- `fix/payment-timeout`

### 示例 3：准备正式发布版本

项目当前版本：`1.2.3`

用户说：

- “帮我准备 1.3.0 版本”

AI 默认建议：

- 创建分支：`release/v1.3.0`

在这个分支上完成版本清单、文档、变更汇总、必要修复。

---

## 不要做的事

1. 不要默认直接在 `main` 上开发
2. 不要在没有检查工作区状态时直接切分支
3. 不要凭空捏造版本号
4. 不要在没验证的情况下假装“已经没问题”
5. 不要未经用户明确要求就直接发 release 或强推

---

## 给你的最简用法

如果你以后想直接让 AI 按这个流程工作，可以这样说：

```text
按 AGENTS.md 的规则处理这个需求。
先检查版本号和测试方式，创建合适的分支，再开始修改。
```

或者更短一点：

```text
按仓库规则开分支、改代码、跑测试。
```

---

## 最终默认约定

只记住这一版最核心规则：

1. 先检查版本号
2. 先创建分支
3. 再改代码
4. 改完跑测试
5. 最后汇报分支名和验证结果

这就是 AI 做“新版本 / 新功能 / bug 修复”时的默认流程。
